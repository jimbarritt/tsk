## `--version` flag

Added `--version` / `-V` to both `tsk` and `tskd`. `tsk` uses clap's built-in version
support (`env!("CARGO_PKG_VERSION")`). `tskd` handles it manually alongside `--help`.
Both report the workspace version from `Cargo.toml`.

## FR-1 — Vim keybindings in TUI

Added scrolling to the TUI thread list:

- `j` / `↓` — scroll down one line
- `k` / `↑` — scroll up one line
- `ctrl-d` — page down (half terminal height)
- `ctrl-u` — page up (half terminal height)
- `gg` — jump to top (two `g` presses detected via `last_g` state flag)
- `G` — jump to bottom
- Mouse scroll down/up (3 lines per tick)

Mouse capture enabled via `EnableMouseCapture` / `DisableMouseCapture` (including panic hook).

Scroll is implemented via `Paragraph::scroll((offset, 0))`. `count_rows()` computes the
total rendered line count (including section title, borders, blank separators) to derive
the scroll ceiling correctly.

Bug fixed during testing: `count_rows()` was not counting blank separator lines between
sections, causing the scroll ceiling to be too low and content appearing stuck.

## FR-3 — Status bar + keybindings popup

**Status bar** — persistent 1-row bar at the bottom of the TUI. Shows the active thread
id + slug and a `? help` hint. Uses `Color::DarkGray` background (ANSI colour, theme-aware).
Layout split: `Constraint::Min(0)` for main area + `Constraint::Length(1)` for status bar.

**Help popup** — `?` toggles a floating overlay listing all keybindings. Any key dismisses
it. Implemented with ratatui's `Clear` widget (clears the background) + `Paragraph` with a
rounded `Block` border. No title on the popup. Uses terminal default colours throughout.

Border consistency fix: `top_border()` and `bottom_border()` were using `fg(Color::White)`
while data row `│` characters used `Style::default()`, causing a subtle brightness mismatch.
Fixed by removing explicit colour from border functions — all border characters now use the
terminal default.

## FR-4 — `tsk thread update`

New CLI command:
```
tsk thread update <id-or-slug> [--slug <new>] [--description <new>] [--priority <BG|PRIO|INC>]
```

All flags optional. If slug changes, the thread directory is renamed on disk and `index.md`
header is updated to reflect the new slug, priority, and description.

New daemon handler: `thread.update`. Extracted `resolve_thread_idx()` helper (resolves by
numeric id or slug) shared by `switch_to`, `update`, `wait`, and `resume`.

New event type: `ThreadUpdatedEvent`.

New e2e tests: description change, priority change, slug change + directory rename, no-op
(no fields), slug collision error, unknown id error, CLI priority validation.

## FR-5 — `tsk thread wait` / `tsk thread resume`

New `ThreadState::Waiting { reason: Option<String> }` variant. Serialises as a nested JSON
object (serde's default for struct variants with `rename_all = "lowercase"`):
```json
{ "state": { "waiting": { "reason": "waiting for PR review" } } }
```
Unit variants (`active`, `paused`) remain flat strings — no custom serde needed.

**State model:**
```
         create
           │
           ▼
       ┌────────┐
┌─────▶│ PAUSED │◀──────────────────────────┐
│      └────────┘                            │
│        │    ▲                              │
│       wait  resume                         │ switch-to
│        │    │                              │ (another)
│        ▼    │                              │
│      ┌─────────┐                      ┌────────┐
│      │ WAITING │◀────── wait ──────────│ ACTIVE │
│      └─────────┘                      └────────┘
│                                            ▲
└──────────────── switch-to ─────────────────┘
```

- `wait` works from Active or Paused
- `resume` always returns to Paused (getting back to Active is an explicit `switch-to`)
- `switch-to` on a waiting thread: clears waiting state, sets Active
- `switch-to` on other threads: waiting threads keep their Waiting state (not forced to Paused)
- `wait` on already-waiting thread → error
- `resume` on non-waiting thread → error

`previous_state` field was designed and then removed — not needed since resume always goes
to Paused.

New event types: `ThreadWaitedEvent`, `ThreadResumedEvent`.

TUI: waiting threads appear in their priority sections (PRIO+INC or BG) alongside paused
threads. `W_STATE` column width bumped from 8 → 9 to accommodate "waiting" (7 chars + 1
space padding).

New e2e tests: wait sets state, wait without reason, wait errors if already waiting, resume
restores to paused, resume errors if not waiting, wait persists across restart, CLI wait,
CLI resume.

## Task model — designed (not yet implemented)

Full design completed for the next delta. See `doc/session-log/tasks-design.md` (captured
in memory). Key decisions:

- `tasks.json` per thread directory
- Fields: `id` (`TSK-0001-0001`), `description`, `state`, `due_by` (ISO 8601), `seq` (manual order integer)
- States: `not-started` → `in-progress` → `done`; any state → `cancelled`; `in-progress` ↔ `blocked` (with reason)
- Commands: `task create`, `task start`, `task block <reason>`, `task complete`, `task cancel`, `task update`, `task list`
- All default to active thread; `--thread <id>` for explicit targeting
- **Diversion** pattern: adding tasks to non-active threads without switching context
- Task relationships / blocking-by deferred
- Priority deferred (needs more design thinking)
